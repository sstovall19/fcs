Development Style Guide

Obviously everyone has their own style of programming, and tools such as perl-tidy can help but here is a list of things that all developers, regardless of language, should adhere to in order to keep the Fonality code base consistent. Some of these are taken from the perl style guide on perldoc.perl.org, others are to be in-line with our current code.

Run perl programs under -w at all times. You may explicitly turn it off using the no warnings pragma. You should always run under use strict.

  • 4-column indent, replace tabs with spaces.
  • Opening curly on next line as keyword (to be consistent with current code).
  • Ending curly should line up vertically with the starting keyword.
  • Space before the opening curly of a multi-line BLOCK.
  • One-line BLOCK may be put on one line, including curlies.
  • No space before the semicolon.
  • Space around most operators.
  • Space around a "complex" subscript (inside brackets).
  • Blank lines between chunks that do different things.
  • Uncuddled elses.
  • No space between function name and its opening parenthesis.
  • Space after each comma.
  • Long lines broken after an operator (except and and or ).
  • Space after last parenthesis matching on current line.
  • Line up corresponding items vertically.
  • Omit redundant punctuation as long as clarity doesn't suffer.
  • Document your code with understandable comments and use Pod formatting in a consistent way.
    • Use C<> for function, variable, and module names. Function is more readabile with parentheses after the name, e.g. function().
    • Use B<> for command names like cat or grep.
    • Use F<> or C<> for file names.
  • Always call a module-level method using the complete package name. e.g. Fap::ModuleName::MethodName
  • Return undef instead of die for methods executed through the web. die will result in an Error 500.
  • Create unit tests using Test::More whenever possible.

Other things to think about:

  • Just because you CAN do something a particular way doesn't mean you SHOULD do it that way. Go for readability and understandability over complexity.
  • Don't muddy the code trying to exit a loop at the top or bottom. Use the last operator in the middle if appropriate, and "outdent" it a little to make it more visible.
  • Don't be afriad to use loop labels. They enhance readability and allow for multi-level lop breaks.
  • Avoid using grep(), map() or `backticks` in a void context, that is, when you just throw away their return values. They all have return values so use them.
  • Variable names should be all lowercase with underscores to separate words. CONSTANTS should be in all caps. Avoid obscure abbreviations to improve readability. e.g. $bundle_license, $ITERATIONS
  • Package names being with a Capital and do not use underscores. e.g. Fap::ConfLoader
  • Use the /x modifier on complex regular expressions to include whitespace.
  • Use here documents instead of repeated print() statements
  • Always check the return codes of system calls.
  • Think about reusability. Generalize your code. Consider writing a module or object class.
